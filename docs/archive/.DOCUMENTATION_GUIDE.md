# Documentation Quick Reference

**For maintainers: How to navigate and update documentation**

## Structure Overview

```
cp-whisperx-app/
├── README.md                          # Project overview (root only)
└── docs/
    ├── INDEX.md                       # Documentation hub
    ├── QUICKSTART.md                  # 5-minute start
    ├── DEVELOPER_GUIDE.md             # Contribution guide
    ├── DEVELOPER_STANDARDS_COMPLIANCE.md
    ├── PROCESS.md                     # Development workflow
    │
    ├── user-guide/                    # End-user documentation
    ├── technical/                     # Architecture & internals
    └── reference/                     # Citations, changelog, license
```

## File Purposes

### Root Level (docs/)
- **INDEX.md**: Main navigation hub - links to all docs
- **QUICKSTART.md**: New user onboarding (5 min)
- **DEVELOPER_GUIDE.md**: For contributors
- **DEVELOPER_STANDARDS_COMPLIANCE.md**: Code standards
- **PROCESS.md**: Development workflow

### user-guide/
- End-user facing documentation
- How-tos, workflows, troubleshooting
- Configuration and usage guides

### technical/
- Architecture and design docs
- Pipeline internals
- Environment-specific details

### reference/
- Changelog (version history)
- Citations (academic references)
- License information

## Updating Documentation

### When adding a new feature:
1. Update relevant user-guide/ doc
2. Add technical details to technical/
3. Update docs/reference/changelog.md
4. Update docs/INDEX.md if new doc created

### When fixing bugs:
1. Update troubleshooting.md if user-facing
2. Update changelog.md with fix
3. Update technical/ docs if architectural

### When changing workflows:
1. Update user-guide/workflows.md
2. Update QUICKSTART.md if affects onboarding
3. Update README.md if changes are major

## Link Maintenance

Internal links use relative paths:
- From root: `docs/path/to/file.md`
- From docs/: `path/to/file.md`
- From subdirs: `../other/file.md`

Validate links periodically:
```bash
# Check for broken links
grep -r '\[.*\](.*\.md)' docs/ | while read line; do
  # Validate each link exists
done
```

## Archive Policy

Archive when:
- Phase/sprint completion docs pile up
- Temporary analysis files accumulate
- Old implementation plans no longer relevant

**Archive location**: `archive/phase-docs-YYYYMMDD-HHMMSS/`

**Never archive**:
- User guides
- Technical documentation
- Reference materials
- Active developer docs

## Quarterly Maintenance

Every 3 months:
1. Review and update changelog.md
2. Verify all internal links work
3. Update screenshots/examples if UI changed
4. Archive outdated phase docs
5. Review technical docs for accuracy

## Adding New Documents

1. Create in appropriate subdirectory
2. Add link to docs/INDEX.md
3. Add link to parent README.md
4. Update this guide if new category

## Style Guide

- Use h1 (#) for title only
- Use h2 (##) for main sections
- Use h3 (###) for subsections
- Include code examples with syntax highlighting
- Add navigation footer: [Home](../README.md) | [Index](INDEX.md)
- Keep lines under 100 characters (readability)

## Don't

- ❌ Create docs in project root (except README.md)
- ❌ Create COMPLETE.md or SUMMARY.md files
- ❌ Duplicate content across files
- ❌ Leave broken internal links
- ❌ Create phase/sprint tracking docs (use issues)
